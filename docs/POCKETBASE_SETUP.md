# PocketBase setup for this panel

Create a PocketBase project, set **`VITE_POCKETBASE_URL`** (web) to your public origin (e.g. `https://pb.example.com`), and point the Android app at the same base URL (`PbApi.setBaseUrl` / prefs; default in code is `http://127.0.0.1:8090`).

All **custom data** lives in **Base collections** (not the built-in `users` auth table unless noted). Names are **exact** — the web `rtdbPb` layer and Android `PbApi` use these slugs.

---

## 1. Built-in **Users** (`users`)

Used for **email/password panel login** and **share-link viewer accounts** (`authPb.ts`).

- Enable **Email/password** auth in PocketBase.
- **Registration**: either disable public sign-up and create your admin user in the admin UI, or allow sign-up only while onboarding (then lock it down).
- Link viewers are created as normal `users` records with random emails (e.g. `link_<deviceId>_...@temp...`).

### Singleton-style collections (fixed record **`id` = `singleton`**)

Create **one** record with id **`singleton`** in each (admin UI: create record and set id to `singleton` if allowed, or import). The web uses `getOne("singleton")` / upsert with that id.

| Collection | Fields (suggested types) |
|------------|-------------------------|
| `forwarding` | `enabled` bool, `number` text |
| `sms_bot_settings` | `forward_sms_to_tg` bool, `bot_token` text, `chat_id` text |
| `telegram_settings` | `bot_token` text, `telegram_user_id` number |
| `notifications` | `new_connections` bool |
| `panel_config` | `admin_name` text, `expiry_date` text, `payload` **json** (merged view) |
| `panel_customization` | `payload` **json** |
| `security_settings` | `payload` **json** (see below) |
| `tg_bot` | singleton; the app reads the **full record**. Easiest: one **json** field `payload` (object). Multi-account sync copies whatever fields exist on the current `singleton` row. |
| `notification_settings` | singleton; reserved / minimal usage in current web code. One **json** `payload` (e.g. `{}`) is enough unless you extend the UI. |

**`security_settings.payload`** is a nested JSON object used by the panel, including (non-exhaustive):

- `otp_on_delete`, `otp_on_login`, `forward_sms_to_tg`, `biometric_login`, `pattern_login`, `totp_enabled`, `otp_method`, …
- `linked_accounts`: map of id → `{ email, password, label, ... }` (stored credentials for multi-account switch)

## 3. API rules (important)

**Web panel** uses the PocketBase JS SDK **after** `users` auth — rules should allow authenticated users to list/read/write the collections above as your threat model allows.

**Gaps you must address:**

1. **Public `/view` links** load `device_links/{deviceId}/aes_key` **before** the viewer logs in. That requires either **guest `View` access** on `device_links` (leaks metadata if ids are guessable) or a **custom PocketBase route / Edge function** that validates `?t=` and returns the key. The stock app assumes anonymous **read** of link records is possible unless you change the flow.

2. **Android `PbApi`** currently sends **no `Authorization` header**. Unauthenticated REST calls only succeed if API rules allow **`@request.auth.id = ""`** for the operations those endpoints perform (create/update on `devices`, `sms_messages`, etc.). For production you should **add auth** (e.g. per-device PocketBase user or a long-lived secret header) and tighten rules.

---

## 4. Realtime

The web subscribes to `pb.collection(...).subscribe("*", …)` for live updates. Enable **Realtime** for the collections you use (PocketBase defaults often allow it for authorized users).

---

## 5. CORS

Add your **panel web origin** (Netlify URL, `http://localhost:5173`, etc.) under PocketBase **Settings → API → Allowed origins**.
